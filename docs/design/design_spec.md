# 와드로그 디자인 시스템 명세

이 문서는 와드로그 Android 앱의 UI/UX 고도화를 위한 디자인 원칙을 정의한다. 실제 Kotlin/Jetpack Compose 구현 코드나 확정된 디자인 토큰을 제공하지 않으며, 이후 Design Phase에서 Material 3 기반 UI를 일관되게 적용하기 위한 기준으로 사용한다.

## 1. 디자인 목표

- 운동 직후에도 빠르게 기록할 수 있는 입력 중심 UI를 만든다.
- 기록 앱답게 날짜, WOD, 결과, 수치 데이터를 명확하게 보여준다.
- 서로 다른 WOD를 단순 우열로 판단하지 않고 정량 요약과 맥락을 중심으로 표현한다.
- ChatGPT 질문지와 GPT 답변은 긴 텍스트를 읽고 복사하기 쉬운 구조로 제공한다.
- 완전 무료, 서버 없음, 로컬 저장, OpenAI API 직접 호출 없음이라는 운영 원칙을 UI에서도 혼동 없이 드러낸다.
- 장식보다 입력 속도, 확인 가능성, 데이터 안정성을 우선한다.

## 2. Material 3 기반 디자인 방향

- Material 3 기본 컴포넌트를 우선 활용한다.
- 색상, 타입, shape, elevation은 Material 3의 역할 체계를 따른다.
- 화면은 Scaffold, TopBar, BottomBar, Floating/Bottom Action 영역을 중심으로 안정적으로 구성한다.
- 상태 표현은 Material 3의 enabled, selected, error, disabled, loading 패턴을 따른다.
- 외부 디자인 라이브러리 추가를 전제로 하지 않는다.

## 3. 시각 원칙

와드로그는 “다크 스포츠 감성”을 지향한다. 단, 앱의 본질은 운동 기록 도구이므로 과도한 장식, 강한 네온 효과, 복잡한 배경 이미지는 피한다.

- 어두운 배경 위에 카드 중심으로 정보를 정리한다.
- 핵심 액션은 붉은 계열 또는 강한 Primary 톤으로 분명히 표시한다.
- 수치 데이터는 대비와 정렬을 통해 빠르게 읽히도록 한다.
- 운동 유형, 결과 유형, Rx 상태는 Chip으로 빠르게 구분한다.
- 위험 액션은 일반 액션과 색상, 위치, 확인 절차로 구분한다.

## 4. 테마 정책

### 4.1 다크 테마

- 기본 경험은 다크 테마에 최적화한다.
- Background는 깊은 차콜 또는 블랙에 가까운 톤을 사용한다.
- Surface는 배경보다 살짝 밝게 하여 카드와 입력 영역을 구분한다.
- Primary는 저장, 추가, 복사 같은 주요 액션에 사용한다.
- Error는 삭제, 초기화, Import 덮어쓰기 등 데이터 손실 위험에만 사용한다.

### 4.2 라이트 테마

- Android 시스템 설정에 따라 라이트 테마를 지원할 수 있어야 한다.
- 라이트 테마에서도 스포츠 감성은 유지하되, 흰 배경 위 과도한 고채도 색 사용은 피한다.
- Surface와 Background의 경계가 명확해야 한다.
- 다크 테마에서만 읽히는 색 조합을 사용하지 않는다.

## 5. 컬러 시스템

실제 색상 값은 구현 단계에서 Material 3 ColorScheme으로 조정한다. 아래 값은 방향 제안이며 코드 수준 토큰으로 확정하지 않는다.

| 역할 | 디자인 방향 |
|---|---|
| Primary | 저장, 추가, 복사 등 핵심 액션. 스포츠 감성을 주는 붉은 계열 또는 선명한 포인트 컬러 |
| Secondary | 보조 액션, 선택 보조, 링크성 액션. 차분한 청록 또는 스틸 톤 |
| Tertiary | GPT, Report, 분석 관련 보조 강조. 보라 계열을 과도하게 지배적으로 쓰지 않는다 |
| Background | 앱 전체 배경. 다크에서는 차콜/블랙 계열, 라이트에서는 낮은 채도의 밝은 회색 |
| Surface | 카드, 시트, 입력 패널. Background와 구분되되 과도하게 떠 보이지 않게 설정 |
| Error | 삭제, 초기화, 복구 덮어쓰기, 저장 실패 같은 위험/오류 상태 |
| Outline | 카드 경계, 입력 필드 경계, 구분선. 낮은 대비로 구조만 잡는다 |
| Success | 저장 완료, 복사 완료, 복구 완료 같은 긍정 피드백. 녹색 계열을 절제해서 사용 |
| Warning | Import 덮어쓰기, 데이터 손실 가능성, 미완성 입력 경고. 황색/앰버 계열 |

## 6. 타이포그래피 원칙

| 용도 | 원칙 |
|---|---|
| 화면 제목 | 현재 위치를 빠르게 이해할 수 있게 짧게 작성한다. 예: 오늘 기록, WOD 작성 |
| 섹션 제목 | 카드 묶음의 의미를 설명한다. 예: 결과, Movement, GPT 답변 |
| 카드 제목 | WOD 제목, 날짜, 주요 기록을 함께 읽을 수 있게 한다 |
| 본문 | 한국어 안내와 WOD 원문을 편하게 읽을 수 있는 크기와 행간을 유지한다 |
| 보조 설명 | 선택 입력, 데이터 부족, 추정 정보 안내에 사용한다 |
| 수치 데이터 | 단위와 값을 분리해서 명확하게 표시한다. 숫자는 정렬과 대비를 우선한다 |

타이포그래피는 사용자의 글자 크기 확대 설정에 대응해야 한다. 좁은 화면에서 제목이 길면 줄바꿈 또는 축약을 허용하되, 저장/삭제 같은 핵심 버튼 문구는 잘리지 않아야 한다.

## 7. Shape 원칙

| 요소 | 원칙 |
|---|---|
| Card | 8dp 안팎의 절제된 radius. 스포츠 기록 도구답게 과도하게 둥근 형태는 피한다 |
| Button | Material 3 기본 shape를 따르되 주요 액션은 충분한 높이와 터치 영역을 확보한다 |
| Chip | 선택 상태가 즉시 구분되어야 하며 긴 용어는 줄바꿈보다 축약을 우선한다 |
| Dialog | 위험 액션 확인에 사용한다. 내용은 짧고 결정 버튼이 명확해야 한다 |
| TextField | 입력 앱의 핵심 요소이므로 경계, 라벨, 오류 상태가 선명해야 한다 |

## 8. Spacing 원칙

| 영역 | 권장 기준 |
|---|---|
| 화면 패딩 | 모바일 기준 좌우 16dp 이상. 작은 화면에서는 정보 밀도를 유지하되 가장자리 터치를 피한다 |
| 카드 내부 여백 | 12-16dp를 기본으로 한다. MetricCard처럼 수치 중심 카드는 더 조밀해도 된다 |
| 섹션 간격 | 섹션 사이 20-24dp 수준으로 정보 묶음을 구분한다 |
| 입력 필드 간격 | 연속 입력은 12dp 안팎으로 빠르게 훑을 수 있게 한다 |
| 하단 액션 영역 | 시스템 내비게이션과 겹치지 않게 안전 영역을 확보한다 |

## 9. Card 디자인 원칙

- Home, Calendar 선택 날짜, WOD Detail, Compare, Prompt, Report에서 카드 중심 정보 구조를 사용한다.
- 카드 하나에는 하나의 명확한 목적만 담는다.
- 카드 내부에 또 다른 카드 형태의 큰 컨테이너를 중첩하지 않는다.
- 수치 요약 카드는 값, 단위, 설명 순서가 일관되어야 한다.
- 빈 카드처럼 보이는 장식용 컨테이너는 피한다.

## 10. Button 디자인 원칙

- 저장, 추가, 복사 같은 주요 액션은 화면 하단 또는 엄지 접근성이 좋은 위치에 둔다.
- 보조 액션은 텍스트 버튼 또는 Outlined 스타일로 주요 액션과 구분한다.
- 삭제, 초기화, Import 덮어쓰기 같은 위험 액션은 Error 색상과 확인 다이얼로그를 사용한다.
- 버튼 문구는 한국어로 짧게 작성한다. 예: 저장, 복사, 삭제, 추가.
- 비활성 상태는 이유를 주변 안내 문구나 오류 문구로 설명한다.

## 11. Chip 디자인 원칙

- WOD Type, Score Type, Rx/Scaled/Custom 선택에 사용한다.
- 선택된 Chip은 배경, 텍스트, outline 변화가 모두 분명해야 한다.
- CrossFit 용어는 영어 표기를 유지한다. 예: AMRAP, For Time, Rx, Scaled.
- 선택지가 많으면 가로 스크롤보다 줄바꿈 가능한 그룹 구성을 우선 검토한다.

## 12. TextField 디자인 원칙

- 필수 입력과 선택 입력을 명확히 구분한다.
- 사용자가 모르는 값은 비워둘 수 있어야 한다는 요구사항을 반영한다.
- 긴 WOD 원문과 GPT 답변 입력은 다중 행 입력으로 제공한다.
- 숫자 입력은 단위 표시를 함께 제공한다. 예: kg, reps, m, kcal, 초.
- 오류 문구는 사용자를 탓하지 않고 해결 방법을 알려준다.

## 13. Dialog 디자인 원칙

- 모든 위험 액션은 확인 다이얼로그를 거친다.
- 삭제, 초기화, Import 덮어쓰기처럼 되돌릴 수 없는 작업은 그 사실을 명시한다.
- 다이얼로그 제목은 짧게 작성한다. 예: 삭제하시겠습니까?
- 기본 포커스는 안전한 선택에 두는 방향을 권장한다.

## 14. 데이터 시각 표현 원칙

- 최근 3회 비교는 정량 요약 중심으로 표현한다.
- 서로 다른 WOD의 Time, reps, load 등을 단순 순위로 비교하지 않는다.
- 동일 WOD 반복으로 판단 가능한 경우에만 직접 기록 비교를 강조한다.
- 수치에는 단위를 항상 붙인다.
- 데이터가 부족하면 “기록이 부족해 가능한 항목만 보여드립니다”처럼 명확히 안내한다.
- GPT 관련 정보는 앱 내부 판단이 아니라 사용자가 ChatGPT에 붙여넣는 질문지와 저장된 답변임을 분명히 한다.

## 15. 접근성 기준

- 최소 터치 영역은 48dp 이상을 목표로 한다.
- 주요 텍스트와 배경의 명도 대비는 WCAG AA 수준을 목표로 한다.
- 작은 화면에서도 저장/추가/복사 액션이 시스템 내비게이션에 가려지지 않아야 한다.
- 글자 크기 확대 시 카드 제목, 버튼 문구, 입력 라벨이 잘리지 않아야 한다.
- TalkBack 사용 시 화면 제목, 입력 라벨, 선택 상태, 오류 상태가 읽혀야 한다.
- Chip 선택 상태는 색상만으로 구분하지 않고 선택됨 상태 설명이 제공되어야 한다.
- 위험 액션 버튼은 TalkBack에서 위험성을 이해할 수 있는 라벨을 가져야 한다.
